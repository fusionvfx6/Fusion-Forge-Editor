

## 1. Diagnóstico Actual

### Arquitectura General
- **Tipo**: Fork directo de Microsoft VS Code (confirmed by package.json repository URL)
- **Estructura**: Monorepo estándar de VS Code con carpetas `src/vs/{base,code,editor,platform,server,workbench}`
- **Build System**: Gulp-based (gulpfile.js delega a ./build/gulpfile)
- **Tipo de Aplicación**: Desktop Electron application (main.ts imports electron, bootstrap-esm.js)
- **Frontend**:
  - Basado en Chromium/Electron renderer
  - Contiene componentes React en `src/vs/workbench/contrib/void/browser/react/` (según package.json scripts)
  - Usa webpack para bundling de código cliente (evidenciado por dependencias dev como webpack, css-loader, etc.)
- **Backend**:
  - Node.js principal proceso (main proceso de Electron)
  - Servicios nativos a través de módulos como `@vscode/sqlite3`, `@vscode/spdlog`, `native-watchdog`, `kerberos`
  - Extensiones hospedadas en procesos separados (ExtensionHost)

### Cómo se ejecuta en desarrollo
- Scripts principales en package.json:
  - `npm run watch`: Inicia modo watch para cliente y extensiones
  - `npm run watch-client`: Observa cambios en cliente
  - `npm run watch-extensions`: Observa cambios en extensiones
  - `npm run compile`: Compilación completa
  - `npm run test-node`: Tests de Node.js
  - `npm run test-browser`: Tests de browser con Playwright
- Requiere: Node.js, Python (para algunos módulos nativos), herramientas de build de VS Code

### Cómo se construye en producción
- Scripts de producción:
  - `npm run compile`: Compila aplicación completa
  - Scripts específicos de plataforma en `./scripts/` (como se indica en documentación)
  - El build produce archivos en carpeta `out/` estructurada como VS Code estándar
  - No hay scripts de "dist" o "package" explícitos en package.json raíz, indicando que el empaquetado se hace a través de scripts internos de VS Code

### Qué arquitectura usa actualmente
- **Arquitectura Electron estándar**:
  - Proceso principal (main): `src/main.ts` -> `src/vs/code/electron-main/main.ts`
  - Proceso de renderizador: Código cliente compilado desde `src/vs/` empaquetado en `out/vs/`
  - Proceso de ExtensionHost: Separado para seguridad
  - Comunicación vía IPC (Node.js pipes y message passing)
- **Capas observadas** (pero no siguen estrictamente hexagonal):
  - Platform: Servicios comunes (`src/vs/platform/*`)
  - Workbench: UI principal (`src/vs/workbench/*`)
  - Editor: Núcleo de edición (`src/vs/editor/*`)
  - Extensiones: Sistema de plugins (`src/vs/extensions/*`)

## 2. Riesgos (Priorizados por Severidad)

### Críticos (Blocking)
1. **Dependencia de infraestructura de build de VS Code**:
   - El sistema de build depende de scripts internos de VS Code en `./build/` y `./gulpfile.*`
   - Archivos críticos: `build/gulpfile.js`, `build/lib/tasks/*.js`, `build/npm/*.js`
   - *Riesgo*: Cualquier cambio en el upstream de VS Code requiere merges constantes

2. **Binarios nativos específicos de plataforma**:
   - Dependencias como `@vscode/sqlite3` (v5.1.8-vscode), `@vscode/spdlog` (v0.15.0), `native-watchdog`, `kerberos`
   - Estos requieren compilación específica para cada plataforma/arquitectura
   - *Riesgo*: Complejidad significativa en cross-compilation y distribución

3. **Servicios de Microsoft aún embebidos**:
   - URLs de telemetry y actualización probablemente todavía apunten a servidores de Microsoft
   - Crash reporting integrado con AppCenter (Microsoft)
   - Galería de extensiones predeterminada: `https://marketplace.visualstudio.com/_apis/public/gallery`
   - *Riesgo Legal*: Distribución no autorizada usando servicios de Microsoft

4. **Branding residual de VS Code**:
   - Aunque hay cambios en product.json, es probable que queden referencias a "Code", "VS Code" en:
     - Recursos internos
     - Mensajes de error
     - Nombres de servicios
     - Claves de registro
   - *Riesgo*: Inconsistencias en experiencia de usuario y problemas legales

### Altos
5. **Electron versión desactualizada**:
   - package.json muestra `"electron": "34.3.2"` en devDependencies
   - Esta versión puede carecer de parches de seguridad importantes
   - *Riesgo*: Vulnerabilidades conocidas en Chromium/Node.js integrados

6. **Falta de canales de actualización definidos**:
   - No se observan scripts de auto-actualización en package.json
   - El código menciona servicios de actualización (`platform/update/common/update.config.contribution.js`)
   - Pero sin configuración clara de endpoints propios
   - *Riesgo*: Imposibilidad de distribuir actualizaciones seguras

7. **Problemas de permisos y sandbox**:
   - Código muestra lógica compleja para manejo de sandbox de Electron
   - Uso de `original-fs` en lugar de `fs` sugiere workarounds para restricciones
   - *Riesgo*: Aplicación que requiere privilegios excesivos o falla en entornos restringidos

### Medios
8. **Dependencias con licencias potencialmente restrictivas**:
   - Necesita revisar todas las dependencias en package.json para compatibilidad con redistribución
   - Algunas como el SDK de Ollama, MistralAI, etc. pueden tener términos específicos
   - *Riesgo*: Violación de términos de servicio de dependencias de terceros

9. **Hardcodeo de rutas y puertos**:
   - Búsqueda inicial muestra referencias a puertos y paths específicos
   - Necesita revisión detallada para asegurar portabilidad
   - *Riesgo*: Fallos en entornos con restricciones de red o estructura de archivos diferente

### Bajos
10. **Documentación de build incompleta**:
    - Falta documentación clara sobre cómo generar instaladores desde cero
    - Dependencia de conocimiento tácito de los scripts de VS Code
    - *Riesgo*: Barrera para nuevos contributors en el proceso de liberación

## 3. Arquitectura Recomendada

### Principios Guía
1. **Separar claramente lo que esVoid de lo que es infraestructura de VS Code**
2. **Mantener compatibilidad con extensiones de VS Code estándar** (punto clave de valor)
3. **Reducir superficie de ataque mediante actualización de Electron y fortalecimiento de sandbox**
4. **Implementar canal de actualización propio independiente de Microsoft**
5. **Mantener capacidad de compilación reproducible y cross-platform**
6. **Definir contratos estables y versionados entre capas (runtime core, extensiones, servicios) para permitir evolución independiente**

### Arquitectura Objetivo
```
[Fusion Forge Desktop Application]
│
├── Main Process (Electron)
│   ├── Servicios del Sistema (actualización, notificación, etc.)
│   ├── Gestor de Ventanas
│   ├── Puente de Seguridad (IPC validado)
│   └── Servicios de Integración (FS, registro, etc.)
│
├── Renderer Process (Chromium)
│   ├── Workbench UI (basado en VS Code pero con branding Void)
│   ├── Contribuidores de Funcionalidad
│   └── Motor de Edición (Monaco fork)
│
└── Extension Host Process (Separado)
    ├── Entorno de Extensiones de VS Code (compatibilidad mantenida)
    ├── Servicio de Lenguaje
    └── Servicio de Depuración
```

### Tecnologías Clave a Mantener
- **Electron**: Como shell desktop (pero actualizado a última versión LTS)
- **Monaco Editor**: Fork interno o upstream con mejoras Void
- **Extensiones VS Code**: Mantener punto de compatibilidad clave
- **Servicios de Plataforma VS Code**: Aquellos no dependientes de Microsoft

### Tecnologías a Reemplazar/Actualizar
- Sistema de build: Migrar de Gulp a npm scripts estándar o herramientas modernas como esbuild/rollup donde sea apropiado
- Servicios de actualización: Implementar sistema propio basado en electron-updater o similar
- Telemetry: Reemplazar con solución propia o desactivar por defecto
- Galerías: Punto de configuración para marketplace propio o alternativo (Open VSX)

## 5. Recomendación Final

### Estrategia Seleccionada: **Modernización Controlada con Mantención de Compatibilidad de Extensiones**

**Justificación Técnica**:

1. **Valor Diferencial Clave**: La principal ventaja de Fusion Forge sobre otros editores es su potencial para ser una distribución de VS Code con capacidades de IA integradas (como se evidencia por dependencias como `@anthropic-ai/sdk`, `@google/genai`, `@mistralai/mistralai`, `ollama`, `openai`, `groq-sdk`). Romper la compatibilidad con el ecosistema de extensiones de VS Code destruiría este valor.

2. **Factibilidad Técnica**: El código base es esencialmente VS Code con modificaciones. Intentar un reescritura completa sería prohibitivamente costoso y arriesgado. La aproximación de modernización incremental permite:
   - Mantener el funcionamiento durante la transición
   - Validar cambios en cada fase
   - Retroceder fácilmente si es necesario

3. **Reducción de Riesgo Legal**: Al reemplazar sistemáticamente los servicios de Microsoft y el branding, se elimina el riesgo de infracción mientras se mantiene la funcionalidad central. Esto es preferible a intentar obtener licencias de Microsoft para redistribuir sus servicios (lo cual es poco probable que concedan).

4. **Sostenibilidad a Largo Plazo**: Al actualizar el sistema de build hacia estándares más modernos (incluso si se mantiene Gulp por compatibilidad inicial) y actualizar dependencias críticas como Electron, se reduce la deuda técnica y se mejora la postura de seguridad.

5. **Viabilidad de Distribución Multiplataforma**: La arquitectura Electron es inherentemente multiplataforma. Al enfocarse en reemplazar los componentes específicos de Microsoft (servicios de update, telemetry, galerías) en lugar de rehacer la arquitectura completa, se aprovecha el trabajo existente de VS Code en soportar Windows, macOS y Linux.

### Por qué NO las otras opciones:

- **Migrar completamente a electron-builder desde cero**:
  - Requiere rehacer todo el sistema de build y empaquetado
  - Alto riesgo de romper funcionalidades complejas de VS Code
  - No aprovecha el trabajo existente de VS Code en manejo de extensiones y lenguajes

- **Crear un híbrido con Tauri/Wails**:
  - Requiriría reescribir la capa de renderer completa
  - Pérdida casi total de compatibilidad con extensiones de VS Code
  - El motor de edición (Monaco) está profundamente integrado con la arquitectura web de VS Code

- **Mantener el sistema de heredado sin cambios**:
  - Riesgo legal continuo por uso de servicios de Microsoft
  - Dependencia de versiones potencialmente inseguras de Electron
  - Imposibilidad de distribuir actualizaciones propias
  - Mala percepción de producto "derivado no autorizado"

### Conclusión Técnica

La estrategia de modernización controlada ofrece el mejor equilibrio entre:
- **Preservación del valor** (compatibilidad con extensiones de VS Code)
- **Reducción de riesgo** (eliminación de dependencias de Microsoft)
- **Factibilidad** (aprovechando el 95%+ del trabajo existente)
- **Sostenibilidad** (base técnica actualizable a largo plazo)

Esta aproximación permite a Fusion Forge evolucionar como una distribución legítima y diferenciada de VS Code, enfocada en capacidades de IA integradas, mientras mantiene acceso al vasto ecosistema de extensiones que hacen a VS Code exitoso.# Technical Audit Report: Fusion Forge (Void) Multiplatform Distribution Strategy
